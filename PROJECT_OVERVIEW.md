# 项目见解：NL2DSL — 自然语言查数助手

## 一句话说清楚

让业务人员用日常中文（"宁巢·东城公寓2月应收环比增长多少"）直接问到数据库里的答案，背后是一个多 Agent 协作的 LLM 系统，生成 Python 代码而不是 SQL 来取数计算。

---

## 它到底做什么

```
业务人员:  "宁巢·东城公寓2月应收金额相较于1月环比增长多少？"
                  │
                  ▼
         ┌────────────────────┐
         │    NL2DSL 系统      │
         │                    │
         │  理解意图 → 抽取槽位 │
         │  → 映射到数据库表字段│
         │  → 生成代码 → 执行  │
         │                    │
         └────────┬───────────┘
                  │
                  ▼
         "2月应收: 610,000元
          1月应收: 580,000元
          环比增长: 5.2%"
```

**输入**：自然语言问题  
**输出**：数据答案（数值 + 计算结果）  
**核心约束**：不直接生成 SQL（安全风险 + 复杂计算难写），生成受控 Python 代码调用安全的 `data_query()` 取数 + pandas 做计算

---

## 为什么需要它

传统查数流程，业务人员要经过：想问题 → 找数据同学 → 数据同学理解需求 → 写 SQL → 跑数 → 返回结果。链条长、沟通损耗大、一个简单查询可能等半天。

NL2DSL 把这个流程压缩成：打字 → 出结果。关键在于它不是把自然语言直接翻译成 SQL 扔给数据库，而是：

- **意图识别**：你想查数？看环比？做归因分析？还是纯闲聊？
- **槽位抽取**：你说的"应收金额"对应哪张表的哪个字段？你说的"东城"对应哪个门店？该问清楚的不会瞎猜
- **代码生成**：把确认好的槽位拼成一段 Python 程序，跑完给你结果

---

## 核心架构：4 个 Agent 协作，各干各的

整个系统的大脑是 **LangGraph Supervisor 多 Agent 模式**，一个总管 + 三个专职工人：

```
                          用户输入
                             │
                             ▼
                   ┌─────────────────┐
                   │  Supervisor     │  "现在该叫谁？"
                   │  (V3, 路由决策)  │
                   └────────┬────────┘
                            │
          ┌─────────────────┼─────────────────┐
          ▼                 ▼                 ▼
   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
   │ IntentAgent  │  │  SlotAgent   │  │  CodeAgent   │
   │ "他想干嘛？"  │  │ "条件是什么？" │  │ "怎么写代码？" │
   │              │  │              │  │              │
   │ 分类：查数/  │  │ 抽实体→打状态 │  │ 槽位+表结构  │
   │ 环比/占比/   │  │ →模糊就反问  │  │ →生成Python  │
   │ 闲聊...      │  │ →确定就往下走 │  │ + DSL 代码   │
   └──────────────┘  └──────────────┘  └──────────────┘
```

拆成 4 个的核心原因：

| 收益 | 说明 |
|---|---|
| **独立评测** | 改了 IntentAgent 的 Prompt，跑完评测只看意图相关指标，不用重测整个系统 |
| **模型按需分配** | 路由和分类用便宜的 V3，代码生成用推理强的 R1 |
| **问题好定位** | Badcase 来了，LangFuse Trace 一看就知道是哪个 Agent 的 output 出了问题 |
| **Prompt 解耦** | 每个 Agent 的 Prompt 只管自己那一亩三分地，不会牵一发动全身 |

### 流程关键细节

```
Supervisor → IntentAgent（分类意图）
     │
     ├── chitchat → Supervisor 直接回复，后面全跳过（省 token）
     │
     └── data_query / mom_growth / ... → SlotAgent（抽槽位）
           │
           ├── 有模糊/未知 → 反问用户 → 等人回复 → SlotAgent 补全
           │
           └── 全确定 → CodeAgent（生成代码）→ exec() 执行 → 返回结果
```

- **闲聊直返**：Supervisor 看到闲聊直接回，完全不跑后面 Agent，几乎零成本
- **反问是 human-in-the-loop**：LangGraph `interrupt()` 暂停等用户回复，不是盲目重跑整条链路
- **多轮继承**：上轮已确认的槽位保留，当前轮只更新被提到的新信息
- **表名在槽位抽取阶段就确定了**：`ALIAS_MAP` 是 `别名 → (表名, 字段名)`，不是到 CodeAgent 才猜

## 关键设计决策

### 1. 生成 Python 而不是 SQL

```
直接生成 SQL                 生成 Python + data_query()
────────────────────        ──────────────────────────
SQL 注入风险                 data_query 内部参数化查询，注入不可能
环比/占比/分组写 SQL 很绕     pandas 几行搞定
中间结果看不到                DataFrame 可以 print 出来
数仓方言不统一                data_query 内部适配，AI 不需要知道
```

### 2. 用受限 exec() 而不是 Docker 沙箱

学习阶段不需要 Docker 的重量。两道防线：Prompt 约束 + `data_query` 表名白名单 + 干净的 `__builtins__`（没有 `open`/`eval`/`__import__`）。进阶了再加 AST 静态扫描和 Docker。

### 3. Schema 检索轻量起步

当前 5-10 张模拟表、总字段不到 50 个，直接把完整表结构 JSON 塞进 CodeAgent 的 Prompt。不需要 BM25、pgvector、Embedding 那一套。等表膨胀到 50+ 再升级到三层混合检索。

## 数据流全景（一次完整查询）

```
┌─────────────────────────────────────────────────────────────┐
│                        一次完整查询                           │
│                                                             │
│  用户: "宁巢·东城公寓的应收金额环比增长多少"                    │
│                                                             │
│  ① Supervisor (V3, ~250 token)                              │
│     分析 State → 路由到 IntentAgent                          │
│                          │                                  │
│  ② IntentAgent (V3, ~900 token)                             │
│     分类 → {labels: ["data_query", "mom_growth"]}           │
│                          │                                  │
│  ③ SlotAgent (V3, ~1500 token)                              │
│     抽取槽位 + 查 ALIAS_MAP 映射到表字段:                     │
│     {                                                       │
│       "metric":    {"table":"ningchao_revenue",             │
│                     "field":"receivable", "status":"确定"},  │
│       "store":     {"table":"ningchao_revenue",             │
│                     "field":"store_name",                   │
│                     "value":"宁巢·东城公寓", "status":"确定"} │
│     }                                                       │
│                          │                                  │
│  ④ CodeAgent (R1, ~2000 token)                              │
│     槽位 + 表结构 Prompt + intent=mom_growth → 生成代码:      │
│     df_curr = data_query({"table":"ningchao_revenue", ...}) │
│     df_prev = data_query({"table":"ningchao_revenue", ...}) │
│     mom = (curr - prev) / prev   ← 算出来的，不是查出来的     │
│                          │                                  │
│  ⑤ exec() 执行代码                                           │
│     data_query → build_sql → 参数化查询 → pd.read_sql       │
│                          │                                  │
│  ⑥ 返回结果给用户                                             │
│     "2月应收: 610,000元, 环比增长: 5.2%"                      │
│                                                             │
│  总 Token: ~4,600  |  成本: ~¥0.04/次  |  延迟: 1-2秒       │
└─────────────────────────────────────────────────────────────┘
```

## 为什么这样设计——核心理念

```
           传统方式                          这个项目
       ┌──────────────┐              ┌──────────────────┐
       │ 一个超大 Prompt │              │ 4 个 Agent 各管一摊 │
       │ 意图+槽位+代码  │              │ 边界清晰、可独立迭代  │
       │ 全混在一起      │              │                    │
       └──────────────┘              └──────────────────┘
              │                               │
              ▼                               ▼
       出了问题全靠猜                       LangFuse Trace
       改了 Prompt                          一眼定位到节点
       不知道哪个环节坏了                    改了谁的 Prompt
                                            看谁的分
```

核心思想就两条：

1. **拆开** — 意图识别、槽位抽取、代码生成是三类完全不同的任务，不该用一个 Prompt 解决。拆成独立 Agent 后，每个 Agent 可以独立调 Prompt、独立评测、独立迭代。

2. **可观测** — 每个节点输入输出全记录（LangFuse），评测指标分节点计算。不是"系统准不准"这种笼统问题，而是"IntentAgent 的归因 Recall 从 0.64 提到 0.78，但闲聊 Precision 掉了 0.04"——改动的效果是定向可计算的。

## 技术栈一句话

| 组件 | 选型 | 理由 |
|---|---|---|
| 编排 | LangGraph Supervisor | 透明可控，学习价值最高 |
| LLM | DeepSeek (V3+R1) | 便宜 + OpenAI 兼容 |
| 结构化输出 | Instructor + Pydantic v2 | 自动 retry + validation |
| 数据库 | Supabase (PostgreSQL) | 同时当元数据库和模拟数仓 |
| 后端 | FastAPI + Uvicorn | 前后端分离，SSE 流式输出 |
| 前端 | Streamlit | 快速原型 |
| 可观测性 | LangFuse | Tracing + Prompt 版本管理 + 实验对比 |

---

## 适用场景

这个架构适合任何 **"自然语言 → 结构化查询 → 返回数据答案"** 的场景：

- 业务人员自助查数（类似 Text-to-SQL 但更安全）
- 内部 BI 助手（"上个月哪个门店营收最高"）
- 报表自动生成（"给我做一张本季度的环比趋势图"）

不适合的场景：开放式对话、创作类任务、需要联网搜索的查询——这些不需要多 Agent 协作取数。
