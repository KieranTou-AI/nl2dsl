# 评测体系 & 成本分析

## 一、Token 用量估算

### 一次典型查询的 Token 消耗

以"查看26年2月宁巢·东城公寓应收金额相较于1月环比增长多少"为例：

| 阶段 | Agent | 模型 | Input | Output | 小计 |
|---|---|---|---|---|---|
| 路由 | Supervisor | V3 | ~200 | ~50 | **~250** |
| 意图识别 | IntentAgent | V3 | ~800 (Prompt+历史) | ~100 | **~900** |
| 槽位抽取 | SlotAgent | V3 | ~1200 (Prompt+表schema) | ~300 | **~1500** |
| 代码生成 | CodeAgent | R1 | ~1500 (槽位+表结构+few-shot) | ~500 | **~2000** |
| **合计（无反问）** | | | | | **~4650** |

如果触发反问（槽位有模糊/未知），新增一轮：

| 额外阶段 | 模型 | Input | Output | 小计 |
|---|---|---|---|---|
| 反问生成 | V3 | ~800 | ~100 | ~900 |
| 用户回复 + SlotAgent 重跑 | V3 | ~1000 | ~200 | ~1200 |
| **反问额外开销** | | | | **~2100** |

两种情况：

| 场景 | Token 消耗 | 占比 |
|---|---|---|
| 无反问（槽位全部确定） | **~4,600** | 70% 的查询 |
| 有反问（1轮澄清） | **~6,700** | 30% 的查询 |
| **加权平均** | **~5,200** | — |

### 月度成本估算（DeepSeek 官方定价）

| 模型 | Input 价格 | Output 价格 |
|---|---|---|
| DeepSeek-V3 | ¥2 / 百万 token | ¥8 / 百万 token |
| DeepSeek-R1 | ¥4 / 百万 token | ¥16 / 百万 token |

按加权平均每次查询 5,200 token 计算（其中 R1 约 2,000 token，V3 约 3,200 token）：

| 模型 | 单次成本 | 占比 |
|---|---|---|
| V3 部分 (3,200 token) | ¥0.016 | 39% |
| R1 部分 (2,000 token) | ¥0.025 | 61% |
| **单次查询总成本** | **¥0.041** | — |

| 日活 | 日查询量 | 日成本 | 月成本 |
|---|---|---|---|
| 开发调试 | 50 次 | ¥2 | ¥60 |
| 内部试用 | 200 次 | ¥8 | ¥240 |
| 正式上线 | 1000 次 | ¥41 | ¥1,230 |

### 省钱策略

| 策略 | 效果 |
|---|---|
| CodeAgent 开发期也用 V3 | 单次成本降到 ¥0.016 |
| 闲聊直接 Supervisor 回复，不跑后续 Agent | 闲聊查询成本接近 0（只有 ~200 token） |
| 多轮续问复用上一轮 slots（checkpoint） | 不需要重跑 IntentAgent，省 ~900 token |
| 表结构从 Prompt 内嵌改成按需检索 | 表超过 50 张后 Prompt 会膨胀，检索方案更省 token |

## 二、效果评测体系

### 评测不是跑一遍看对不对

PM 视角下，评测需要回答三个问题：
1. **这个版本比以前好吗？**（对比基线）
2. **好在哪，差在哪？**（分指标拆解）
3. **线上用户实际感受如何？**（Badcase 驱动迭代）

### 评测集设计：4 桶

评测集不是随便收集的问题列表。每一桶有明确的评估目标：

| 桶 | 评估目标 | 样例 | 数量建议 |
|---|---|---|---|
| **单一意图** | 基础分类能力 | "昨天宁巢明石公寓的营收是多少" | 100+ |
| **复合意图** | 多标签同时命中 | "宁巢各门店本月应收及环比情况，画个图"（查数+环比+作图） | 60+ |
| **多轮续问** | 上下文承接 | 第1轮"2月应收多少"→第2轮"环比呢" | 40+ |
| **边界 & 脏数据** | 鲁棒性 | "你好"、"最近怎么样"、错别字、超长输入 | 30+ |

评测集的来源优先级：

```
1. 线上点踩数据        ← 最高优先级，真实 Badcase
2. 业务方反馈          ← 实际使用中的痛点
3. 产品自己构造        ← 覆盖边界条件
4. LLM 自动生成        ← 快速扩量，人工抽检
```

### 评测指标：分节点独立评估

不只看一个总分。每个节点有自己的指标，改了谁的 Prompt 就看谁的分。

#### IntentAgent

| 指标 | 计算方式 | 说明 |
|---|---|---|
| 主意图正确率 | 预测标签 ⊇ 期望标签 ? 1 : 0 | 核心指标——至少该命中的都命中了 |
| 各标签 Precision | TP / (TP + FP) | "环比"标签是不是总被误触发？ |
| 各标签 Recall | TP / (TP + FN) | "归因"标签是不是总漏？ |
| 各标签 F1 | 2×P×R/(P+R) | 综合指标 |
| 多轮承接准确率 | 续问场景下意图是否承接正确轮次 | 单独看多轮桶 |

PM 看这个能知道：改了 Prompt 之后，"归因"的 Recall 从 0.6 提到 0.85，但"闲聊"的 Precision 从 0.95 掉到 0.88——**改动的影响是定向的，不是玄学**。

#### SlotAgent

| 指标 | 计算方式 | 说明 |
|---|---|---|
| 实体识别 P/R/F1 | Token 级别的交并比 | 实体有没有抽全 |
| 字段映射准确率 | 映射正确数 / 总实体数 | "应收金额"有没有映射到对的字段 |
| 字段值映射准确率 | 值正确数 / 总实体数 | "宁巢东城"有没有匹配到正确值 |
| 状态判定准确率 | 状态正确数 / 总实体数 | 确定/模糊/未知 有没有判错 |
| 完整槽位 Exact Match | 整个 slots JSON == 期望 ? 1 : 0 | 最严格——全对才算对 |

状态判定的重要性：把"确定"误判为"模糊"→ 多一次不必要的反问 → 用户体验差。把"模糊"误判为"确定"→ 生成错误查询 → 数据错。

#### CodeAgent

| 指标 | 计算方式 | 说明 |
|---|---|---|
| 语法正确率 | 代码能通过 `exec()` 不报错 | 基本门槛 |
| 逻辑正确率 | 人工/LLM 评审逻辑是否符合槽位 | 核心指标 |
| 结果正确率 | 最终输出的数值是否与期望一致 | 终极指标，但依赖模拟数据准确性 |
| SQL 安全通过率 | `data_query` 白名单校验通过 | 不该出现 INSERT/DELETE |

### 评测流程

```
                    每次改 Prompt / 改模型 / 改参数
                              │
                              ▼
                    ┌──────────────────┐
                    │  跑全量评测       │
                    │  python -m src.  │
                    │  eval.runner     │
                    │  --all-buckets   │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌──────────────────┐
                    │  生成评测报告     │
                    │  - 总分 vs 基线  │
                    │  - 各节点分指标  │
                    │  - 退化的 case   │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
         [提升]          [持平]          [下降]
         合入 main       审视变更        回滚 / 分析退化case
                         是否值得
```

### 评测报告模板

每个版本跑完评测后生成一份简短报告：

```markdown
## 评测报告 v0.2.0 | 2026-06-24

### 变更
- SlotAgent Prompt 新增 3 个 few-shot 示例
- IntentAgent 强规则兜底加了一条："分析原因" → keydriver

### 总分对比

| 指标 | v0.1.0 (基线) | v0.2.0 | 变化 |
|---|---|---|---|
| 主意图正确率 | 89.2% | 91.5% | +2.3% ↑ |
| 完整槽位 EM | 72.1% | 73.8% | +1.7% ↑ |

### IntentAgent 分指标

| 标签 | Precision | Recall | F1 | vs 基线 |
|---|---|---|---|---|
| data_query | 0.94 | 0.96 | 0.95 | +0.01 |
| mom_growth | 0.88 | 0.82 | 0.85 | +0.03 |
| keydriver | 0.75 | 0.78 | 0.76 | +0.12 ↑ |
| chitchat | 0.97 | 0.99 | 0.98 | 持平 |

### 退化 Case（3 条）
- #42: "营收怎么样" → 漏标 mom_growth（期望也标）
- #78: 多轮续问第三轮承接了第一轮而非第二轮
- #91: ...

合入。keydriver Recall 提升明显，3 条退化 case 不影响主流程。
```

### 线上监控（上线后）

评测集只能覆盖已知问题。上线后需要：

| 监控项 | 方式 |
|---|---|
| 反问率 | 如果突然飙升 → Prompt 退化或新增用户群体用词不同 |
| 用户点踩率 | 点踩的问题自动入评测集（最高优先级） |
| 代码执行失败率 | CodeAgent 生成了跑不起来的代码 → 需要加 few-shot |
| 平均 Token 消耗 | 异常波动 → 可能是某个 Agent 的 Prompt 被意外撑大了 |

## 三、迭代节奏

PM 视角下的迭代循环：

```
线上 Badcase / 点踩
        │
        ▼
  Badcase 入评测集（写期望输出）
        │
        ▼
  分析：是哪个 Agent 的问题？
  - Intent 分错了？   → 改 IntentAgent Prompt
  - 槽位映射错了？   → 改 SlotAgent Prompt + 补字段别名
  - 代码生成错了？   → 改 CodeAgent Prompt + 加 few-shot
        │
        ▼
  跑全量评测 → 确认未退化 → 上线
        │
        ▼
  看线上数据 → 下一个循环
```

每个 Agent 独立评测 + 独立 Prompt 的价值就在这里：**问题定位精确到一个节点，改完只影响该节点的指标，不会出现"修了一个 bug 引入三个新 bug"的情况。**

## 四、用什么工具落地

上面说的"定位到节点"和"改 Prompt 看分数"不是概念——有现成的开源工具。

### 推荐：LangFuse（开源，一个工具解决两个问题）

**LangFuse** 是一个开源的 LLM 可观测性平台，两个核心能力正好对应你的两个问题：

| 你的问题 | LangFuse 的对应功能 |
|---|---|
| Badcase 怎么定位到节点？ | **Tracing** — 每条请求的完整调用链可视化 |
| 改了 Prompt 怎么看分数？ | **Prompt Management + Experiments** — Prompt 版本化 + 跑分对比 |

不需要自己写日志系统，LangGraph 天然支持把 trace 发到 LangFuse。

### Badcase 定位：Tracing（调用链可视化）

一条用户查询进来，LangFuse 自动记录每个 Agent 的输入/输出/耗时/Token：

```
用户: "2月应收环比增长多少"
        │
        ▼
┌─────────────────────────────────────────────┐
│  LangFuse Trace 视图                        │
│                                             │
│  SupervisorAgent    200ms  · 250 tokens     │
│  ├─ input:  {...}                           │
│  └─ output: next=IntentAgent               │
│                                             │
│  IntentAgent        300ms  · 900 tokens     │
│  ├─ input:  {query, history}               │
│  └─ output: {labels: ["data_query",         │
│              "mom_growth"], ...}      ← 看这里！│
│                                             │
│  SlotAgent          500ms  · 1500 tokens    │
│  ├─ input:  {query, intent, ...}           │
│  └─ output: {slots: [{应收金额: ...}]}      │
│                                             │
│  CodeAgent          800ms  · 2000 tokens    │
│  ├─ input:  {slots, schema}                │
│  └─ output: "df_current = data_query(..."  │
│                                             │
└─────────────────────────────────────────────┘
```

Badcase 来了，打开这条 trace → 从上往下看每个节点的 output → 哪个节点的输出不符合预期，问题就在哪个节点。**不用 grep 日志，不用 print 调试。**

PM 不需要自己部署，研发把 LangFuse 地址给你，你就能在浏览器里看每条请求的完整链路。

### Prompt 迭代：版本管理 + 实验对比

改了 IntentAgent 的 Prompt 之后，怎么知道效果变好了还是变差了？

```
┌──────────────────────────────────────────┐
│  LangFuse Prompt Management              │
│                                          │
│  intent_agent_prompt                     │
│  ├─ v1 (2026-06-20)  ← 基线             │
│  │   主意图正确率: 89.2%                 │
│  │   keydriver F1:  0.64                │
│  │                                       │
│  ├─ v2 (2026-06-22)  ← 加了2条归因few-shot│
│  │   主意图正确率: 90.1%  +0.9%          │
│  │   keydriver F1:  0.76   +0.12 ↑      │
│  │   chitchat P:    0.91   -0.04 ↓      │
│  │                                       │
│  └─ v3 (2026-06-24)  ← 当前生产         │
│      主意图正确率: 91.5%                 │
│      keydriver F1:  0.78                │
│                                          │
└──────────────────────────────────────────┘
```

流程：改 Prompt → 标记为新版本 → 跑评测集 → LangFuse 自动对比新旧版本的指标 → 一目了然。

### 为什么选 LangFuse 而不是 LangSmith

| 对比 | LangFuse | LangSmith |
|---|---|---|
| 开源 | ✅ 可以自己部署 | ❌ SaaS 付费 |
| Tracing | ✅ | ✅ |
| Prompt 版本管理 | ✅ | ✅ |
| 评测跑分 + 对比 | ✅ Experiments | ✅ |
| 学习成本 | 低，Docker 一键部署 | 低，但付费 |
| 适用场景 | 学习项目 + 小团队 | 公司有预算 |

对学习项目，LangFuse 开源自部署就够了。以后公司用 LangSmith 也不冲突——两者功能接近，切换成本低。

### 最小化接入（研发 30 分钟搞定）

```python
# 在 LangGraph workflow 里加两行
from langfuse.callback import CallbackHandler

langfuse_handler = CallbackHandler(
    public_key="...",
    secret_key="...",
    host="http://localhost:3000"  # 自部署地址
)

# 跑 workflow 时传进去
result = app.invoke(
    {"user_query": "..."},
    config={"callbacks": [langfuse_handler]}
)
```

跑完就能在 LangFuse 界面看到这条请求的完整 trace 了。
